ThreadLocal为解决多线程的并发问题提供了一种新的思路，使用这个工具可以很简洁地实现多线程环境下变量共享的问题。

ThreadLocal

当使用ThreadLocal维护变量时，ThreadLocal为每个使用该变量的线程提供独立的变量副本，所以每一个线程都可以独立地改变自己的副本，而不会影响其它线程所对应的副本。实现逻辑很简单，就是维护一个Map，以线程Id作为key，value就是要共享的变量。所以每个线程都可以独立地改变属于自己的变量副本，而不影响其他线程。

这就解释了Local的含义，线程就好像使用本地变量一样，使用共享变量。

和同步机制的比较

ThreadLocal和线程同步机制都是为了解决多线程中相同变量的访问冲突问题。

在同步机制中，通过对象的锁机制保证同一时间只有一个线程访问变量。这时该变量是多个线程共享的，使用同步机制要求程序慎密地分析什么时候对变量进行读写，什么时候需要锁定某个对象，什么时候释放对象锁等繁杂的问题，程序设计和编写难度相对较大。

而ThreadLocal则从另一个角度来解决多线程的并发访问。ThreadLocal会为每一个线程提供一个独立的变量副本，从而隔离了多个线程对数据的访问冲突。因为每一个线程都拥有自己的变量副本，从而也就没有必要对该变量进行同步了。ThreadLocal提供了线程安全的共享对象，在编写多线程代码时，可以把不安全的变量封装进ThreadLocal。

可以这样理解，ThreadLocal是空间换时间的方式，而同步机制是使用时间换空间的方式。二者达到同样的目的。这里的时间是指cpu时间，当然cpu时间是很贵的，所以可以根据代码逻辑需求来更加合理地选择使用哪一种方式来保证变量访问不冲突。

实践经验

ThreadLocal是解决线程安全问题一个很好的思路，它通过为每个线程提供一个独立的变量副本解决了变量并发访问的冲突问题。在很多情况下，ThreadLocal比直接使用synchronized同步机制解决线程安全问题更简单，更方便，且结果程序拥有更高的并发性。

  1. 将ThreadLocal设置为static静态属性，并赋予初始值
  2. 关于ThreadLocal的操作和该静态属性封装于同一个类中，对ThreadLocal的操作可以不用是静态的，每个线程独有不共享
  3. 需要使用的时候，new一个实例，调用实例中的方法，即可方法使用存放在ThreadLocal中的变量
