Flume源码-工作流程

现在工作中主要负责日志实时收集，使用flume和团队自主研发的log_agent、log_agent_file配合完成。目前该套设计方案在资源充分的前提下，具有日均日志条数900亿条，且99%在3s内的数据收集实时保证。这里的3s内是指业务从打印日志到日志落地到kafka的时间。本文主要介绍在看flume源码时，总结的flume的工作流程，flume的版本为1.4.0。版本比较旧，目前开源社区的版本为1.7.0。版本较旧的原因是比起关注新版本功能的完善，更关注服务稳定性，另一方面，我们对1.4.0版本进行了自定义的修改，是服务更贴近业务的需求。

flume基本架构


flume组件主要分为source、channel、sink。source负责收集数据，Channel主要负责缓存数据，sink主要负责消费数据。这里不再赘述。详情可以参见官网介绍flume官网

生命周期管理

flume组件生命周期相关的代码在flume-ng-core路径的lifecycle的子目录下

flume除了monitor和service之外，都具有生命周期的概念，主要作用是用来标记组件当前的状态。flume组件的状态有四个：IDLE,START,STOP和ERROR

flume组件类需要实现LifecycleAware接口，flume对组件初始化完成后会调用LifecycleSupervisor.supervise()将组件加入监控，LifecycleSupervisor类内部对每一个组件会启动一个定时线程MonitorRunnable,在定时线程中调用组件的start和stop函数对组件进行控制。其中stop方法是在unsupervise中调用的，而不通过MonitorRunnable控制。

flume启动流程

flume的main函数在Application类中，该类在flume-ng-node目录下。
flume启动流程主要做了两件事：

  1. 通过配置文件构造组件，连接有关联的组件，将组件填入到MaterializedConfiguration类中
  2. 从MaterializedConfiguration类中获取组件，将组件注册到LifecycleSupervisor中，通过LifecycleSupervisor启动

main函数依次做了以下几件事:

  1. 解析命令行，主要通过org.apache.commons.cli.Options实现，这里不再赘述。
  2. 寻找命令行里的配置文件路径，解析配置文件，并放在一个File对象里
  3. 调用AbstractConfigurationProvider实例，传入File配置文件构造对象，返回一个MaterializedConfiguration对象，该对象里存放所有的sink,sinkgroup,source,channel组件
  4. 调用application.start(),该函数将所有组件加入生命周期管理并注册启动monitor服务
  5. 向JVM注册一个函数，当进程结束时停止所有组件。如：Runtime.getRuntime().addShutdownHook(new Thread())

注：这里要注意第三步其实根据命令行中的no-reload-conf参数【是否关闭重载配置文件】选择调用AbstractConfigurationProvider的派生类PollingPropertiesFileConfigurationProvider或者PropertiesFileConfigurationProvider。

这里详细解说一下AbstractConfigurationProvider,这是一个抽象类，解析配置文件并初始化组件在该类中实现。AbstractConfigurationProvider的作用是初始化所有组件。挂好相互关联的钩子并将组件封装到MaterializedConfiguration对象中返回给上层。AbstractConfigurationProvider类的功能很好理解，关键在于flume支持配置重载，配置重载则channel组件涉及到是否复用的问题【重新生成channel可能会导致之前数据的丢失问题】。AbstractConfigurationProvider类的getConfiguration()函数主要处理了这一套逻辑。

getConfiguration()函数的执行基本流程如下：

  1. 加载配置文件
  2. load channels
  3. load sources，这一块要传入2中load的channel，因为要挂channel和source之间的钩子【这一块实际上是将channel封装到ChannelProcessor中，将ChannelProcessor传入Source，关于ChannelProcessor的作用以后的文章将会介绍
  4. load sinks，这一块要传入2中load的channel，因为要挂channel和sink之间的钩子
  5. 移除没有sink和source连接的channel【如果只有source/sink这里不会检测】
  6. 将sink,source,channel封装在MaterializedConfiguration对象中并返回，注意source是封装在SourceRunner类中，sink是封装在SinkRunner中。这里的定义是Runner负责与操作系统打交道，而source/sink专注于维护自身逻辑。SinkRunner中还要注意一点是一个Sink只能在一个SinkGroup里面。

有几个需要注意的点：

  1. 加载配置文件时调用了FlumeConfiguration类，该类内部有对配置文件进行合法性校验，校验的错误列表可以通过getConfigurationErrors()接口获取
  2. 每个组件都有对应的Factory工厂类（如ChannelFactory），通过调用Factory类的create()函数，根据配置文件利用反射创建对象。故扩展自定义组件时，可以在type中设置类的全路径名
  3. load_channel函数中channel可以通过加上@Disposable声明表示不复用，重新加载配置时生成一个新的channel
  4. 细读load_channels,load_sources,load_sinks代码，可以发现加载组件时分了两种情况，has ComponentConfiguration object和do not hava a ComponentConfiguration object。这一块可以不用理会，细读配置文件逻辑好像全部都是后一种do not hava a ComponentConfiguration object

以上步骤执行完，flume就可以提供服务了。
